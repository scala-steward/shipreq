Let's start with the basics. We need to store the draft somewhere as the user types. We needn't save on every single keystroke (for performance reasons), but there should be some kind of window., likely temporal. Another way of looking at that window, is how much user change are we happy to lose in the case of disaster? 10, 20, 30 sec worth? Kind of arbitrary so just use a gut-feel to compromise between client performance, server impact, and data protection for the user. Don't forget that we'll also save on certain events like editor blur, tab closing beacon.
How? My first thought was local storage but there are some issues with that:
  • security risk -- we're talking about IP here although it's only a little I guess, plus encryption could be employed to reduce risk
  • drafts are bound to a single browser cache - that's pretty shitty for users. Users should be able to switch devices and access their drafts
Meaning that we're going with a server-side solution. What options do we have?
  • DB - just chuck it all in Postgres
  • setup some other data store?
I guess the only reason not to use Postgres is for performance reasons right?
  • more svr ↔ DB traffic
  • more DB storage
  • table performance - there's going to be A LOT of frequent writes (eg. potentially approx every n sec per concurrent user)
Oh shit there's something else to consider: how do we save drafts when a user is offline? It's a scenario that we #must support for this to be truly useful, and keep in mind that [MF-36: Offline mode] is going to be a big selling point in the future.
So it looks like we've have two stores:
  1. local storage that syncs itself up when an online connection is restored
  2. a server-side store capable of lots of RW IO

Conflicts

Oh dreaded conflicts. It's going to happen. We need to deal with this for both this, for [MF-36: Offline mode], and for [MF-9: Collaboration: authoring (aka multi-user)].
Let's with a bit of modelling (for a single editor) to see what we're dealing with...
  • client
    • editor: ∅ | V
    • local storage: ∅ | V
    • event received: ∅ | V
    • event sent: ∅ | V
    • draft sent: ∅ | V
    • draft received: ∅ | V
  • server
    • store: ∅ | V
Hmm looking at it that way it doesn't seem that complicated (so long as we have a sense of a time), maybe we just need sensible propagation logic and a single conflict editor.
It would be good to automatically resolve trivial conflicts without the user being notified.
Otherwise we need to popup a dialog for the user to deal with it. A critical point is that it #must not be a forceful modal. The user should still maintain full control over their app's state, meaning a few things:
  • if the user is editing A and a conflict on B occurs, they users' workflow should not be affected. They should maybe see a little conflict count increase at the top of the screen (or something like that if anything at all), be able to finish what they're doing with editor A, then in their own time check out what the story is with B.
  • when a user has the conflict editor open, they should be able to close it and come back to it later

Accidental closure

Going back to the original feedback, we also need to protect users from their own mistakes. If I close an editor and then realise "oh shit, that was the wrong editor/decision", how can I get my old value back? Keep in mind that closing and editor to deliberate discard your changes and then opening it back up again is a very common ability that must be preserved. Even if we separated the "close editor" action from the "discard draft", it's the same problem in that how do we handle the user explicitly discarding their draft and then realising that was the wrong move? There's only so much protection we can provide (or at least is feasible for me to implement right now)... #TODO{Try out a few UCs for this...}

What does Gmail do

What does gmail do? It keeps drafts as separate entities that are manually discardable. Our situation is a bit different. We don't want floating, separate drafts. A draft is associated with exactly one editor — every where the editor goes, so too should the draft. We could associate multiple drafts with a single editor but that seems stupid. I imagine it would just be a proxy for users to manually manage changes they've made on different devices with using the conflict editor. Actually...

History UX per field

...as weird as it sounds that could be a path if you consider that editors always have multiple versions (i.e. it's committed history plus any drafts). I wonder if it would make sense to solve draft management and field history in the same stroke? We'd still need a conflict editor though because regardless of where the two states come from, we don't want users having to manually copy-paste to merge themselves with no automated support. Holy shit, this could actually turn into a real solution. Come to think of it, all of the above is written as if there are always, ever only two revisions needing a merge. What happens when a user has a local draft, a remote draft and a new event comes in? Now there are three sources. If we're saying that merges can be postponed does that mean there can also be an unsaved live change in the editor too, being a fourth source?
Interestingly, this might solve the accidental closure issue: even when a user discards a draft, just keep a non-current draft available in the remote store that the user can view in the history. (?)
