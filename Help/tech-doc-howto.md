# How to write technical documentation:

**Scope and purpose of documentation**

**Scope** - features and uses of ShipReq. common workflows.

**Purpose**

1. To allow users to look up a particular feature
2. To list common ways of using shipreq - workflows
3. To give user information to be able to customise shipreq to their particular needs


**Goals**

* Allow users to look up a particular feature for details on how to use it. E.g. tags, Req Table, etc.
* Give the users an overview of "concepts behind shipreq" - e.g. forgiving of failure, fast, doesn't lose data on crash, etc.
* Allow users to quickly look up shortcuts associated with various features
* Allow users to look up customisations within each feature
* Give users a step by step guide to creating their first project
* Give users a walkthrough on how to effectively use shipreq for various roles
  * Developer
  * Project manager
  * Business analyst
  * Other stakeholders

### **Notes on how to write documentation**

* create a style guide.
  * How will it look?
  * hierarchy of information presented should be consistent
  * what level of language and wording will be used?

More notes:

1. Write descriptive titles. E.g. How do you add a new team member? How do I create a requirement?
2. Use annotated screenshots (pic worth 1000 words)
3. Include links to related articles/details
4. Organise articles into logical groups - table of contents
5. Make doc easy to search
6. Include the end result in the doc page. - screenshot with successful condition.
7. Show steps and substeps
8. Use unique URLs for each doc page - so you can easily share with users.
9. Don't mix conceptual information with instructional information.
10. Include synonyms in index/search feature
